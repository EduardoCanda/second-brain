# S3 (Simple Storage Service)

## O que é
Armazenamento de objetos durável e escalável.

## Caso de uso
Data lake, backup, mídia estática, artefatos de pipeline e logs.

## Por que existe
Existe para armazenar grandes volumes sem gerenciar filesystem/servidor.

## Trade-offs
- ✅ Excelente durabilidade e custo competitivo.
- ❌ Não substitui filesystem POSIX e semântica de bloqueio local.

## Boas práticas
- Use classes de storage e lifecycle policies.
- Habilite versionamento e criptografia por padrão.
- Trate permissões de bucket com extremo cuidado.

## Quando não usar
- Quando o problema pode ser resolvido com uma opção mais simples e barata.
- Quando o time não tem maturidade operacional para sustentar a complexidade do **S3**.
- Quando os requisitos de latência, compliance ou portabilidade pedem outra estratégia.

## Erros comuns
- Escolher tecnologia por hype, sem mapear padrão real de acesso/tráfego.
- Ignorar custo total (execução + transferência + observabilidade + operação).
- Não definir limites, alarmes e dono do serviço em produção.

## Checklist de decisão rápida
- Qual SLO/SLA esperado (latência, disponibilidade, throughput)?
- Qual o volume de tráfego e perfil de crescimento (picos previsíveis ou não)?
- Qual o nível de esforço operacional aceitável para o time?
